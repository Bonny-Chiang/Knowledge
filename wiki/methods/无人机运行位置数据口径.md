# 无人机运行位置数据口径

## Purpose

本方法页用于区分政务无人机数据中的不同位置字段，避免把“任务发生地”“起飞点”“降落点”“案件现场”“飞行轨迹”“公开脱敏点位”混为一谈。

## Key Answer for PPB UAS

Portland Police Bureau UAS dashboard 里的 location 主要应理解为 UAS deployment 关联的 call location / block address，即警务 call for service 的泛化地点。

它不是无人机起飞位置，也不是降落位置，也不是完整飞行轨迹。

在 2025 年报告中，DFR pilot 另行说明无人机被 pre-positioned at a fixed launch site at East Precinct。这说明 launch site 是单独的基础设施位置变量，不是 dashboard 中每条 UAS call 的 location 字段。

## Location Types

### 1. Call Location

任务请求或服务请求的地点。

在 PPB UAS dashboard 中，公开地图展示的是 call location 的泛化点位。它适合分析 UAS deployment 与警情类型的空间分布。

### 2. Incident or Scene Location

案件、事故、可疑物品、逃逸嫌疑人、搜救对象等任务对象所在地点。

它可能与 call location 接近，但不一定完全相同。例如 call may originate from a residential property，而 suspect later moves to another property, rooftop, garbage bin, creek bed, or containment area。

### 3. Launch Site

无人机起飞或预置地点。

DFR 场景中 launch site 尤其重要，因为起飞点会影响 response time、覆盖半径、facility needs、staffing 和 infrastructure cost。PPB 2025 报告将 East Precinct fixed launch site 作为 DFR pilot 条件单独说明。

### 4. Landing Site

无人机返航或降落地点。

年度报告和 dashboard 没有把 landing site 作为公开透明字段重点披露。实际管理中，landing site 可能与 launch site 相同，也可能因任务、故障、现场条件而不同。

### 5. Aircraft Track

无人机飞行轨迹，包括起飞后经过的路径、高度、速度和时长。

这类数据与空域监管、安全审计和隐私保护高度相关，但 PPB 年度报告的公开 dashboard 不是轨迹数据集。

### 6. Sensor Footprint

相机、热成像、红外或其他传感器实际覆盖到的空间范围。

这比 call location 更敏感，因为它关系到谁或哪些地块被观察、记录或保存影像。

### 7. Public Generalized Location

对外公开时经过脱敏、泛化或隐藏的位置。

PPB dashboard 使用 100-block / nearest intersection / generalized latitude-longitude 的方式释放地点。敏感案类可能不显示位置。这类点位适合透明度和宏观分布分析，不适合精确空间建模。

## Interpretation Rules

- 看到 `location` 时，先问它是 call location、incident location、launch site 还是 aircraft track。
- 公开 dashboard 的点位通常已经脱敏，不能当作精确地址。
- DFR 研究必须把 launch site 单独建模；不能用 call location 反推起飞点。
- 热区分析可使用 call location，但起降点选址必须另用 launch-site / candidate-site 数据。
- 绩效评价中的 response time 需要同时记录 dispatch time、launch time、arrival on scene 或 first overwatch time。
- 隐私评估应关注 sensor footprint 和 data retention，而不只是 call location。

## Standardization Implication

政务无人机运行数据标准至少应拆分以下空间字段：

| Field | Meaning | Example Use |
| --- | --- | --- |
| `call_location_generalized` | 任务请求泛化地点 | 公开透明 dashboard |
| `incident_scene_location` | 案件或事故现场 | 证据、处置、复盘 |
| `launch_site_id` | 起飞点或预置点 | DFR 覆盖、资源配置 |
| `landing_site_id` | 降落点 | 安全管理、设备回收 |
| `aircraft_track` | 飞行轨迹 | 空域合规、风险审计 |
| `sensor_footprint` | 拍摄或感知覆盖范围 | 隐私评估、数据安全 |
| `public_location_release_level` | 公开粒度 | 脱敏、共享、安全 |

## Application to Current Projects

对 [[低空政务巡查空间识别方法]] 而言：

- 任务热区识别可使用 call / incident location。
- 起降点布局不能只用 call location，需要候选 launch site、道路可达性、电力接入、禁飞区和净空条件。
- 飞行效率评价需要 launch site 与 incident location 之间的响应关系。
- 数据安全标准要覆盖 sensor footprint 和影像留存，而不是只管点位。

对 [[政务无人机运行数据标准化政策建议]] 而言，PPB 案例说明“公开透明”与“精确运行管理”需要两套粒度：公开数据可以泛化，内部治理必须保留可审计的精细字段和访问控制。
